# Аудит архитектуры проекта Eltera

Проведен анализ ключевых файлов frontend-приложения Eltera (`app.js`, `render.js`, `dashboard-components.js`, `dashboard-premium.js`, `scoring.js`). Ниже представлен разбор текущего состояния кодовой базы с выделением критических проблем и рекомендаций по улучшению.

## 1. Архитектурные проблемы и технический долг

### 1.1. Монолитная обработка событий (`app.js`)
Файл `app.js` содержит более 1500 строк, из которых половина — это глобальные делегированные обработчики событий (`document.addEventListener("click", ...)`). 
- **Проблема:** Бизнес-логика (расчеты, создание ссылок, обновление баланса), управление состоянием (`state.modal = ...`) и побочные эффекты (создание DOM-элементов для toast-уведомлений) смешаны в одних и тех же функциях.
- **Риск:** Высокая вероятность регрессий при добавлении новых фич. Отсутствие изоляции затрудняет тестирование.

### 1.2. Дублирование UI-компонентов
В проекте существует две параллельные системы компонентов:
- Старые компоненты (`KpiCard`, `ChartCard`, `Heatmap`, `AttentionPanel`) в `dashboard-components.js`.
- Новые "премиум" компоненты (`PremiumKpiCard`, `PremiumChartCard`, `PremiumHeatmap`, `PremiumAttentionPanel`), которые частично находятся в `dashboard-components.js`, а частично дублируются в `dashboard-premium.js`.
- **Проблема:** Разный дизайн-код (например, использование префиксов `elt-` в новых компонентах и их отсутствие в старых), дублирование логики рендера графиков и статусов.

### 1.3. Огромный файл рендеринга (`render.js`)
Файл `render.js` (почти 2400 строк) содержит рендер всех страниц и сложных модальных окон (например, мастера оценки).
- **Проблема:** Шаблонные строки HTML перемешаны с логикой ветвления. Нет разбиения на модули по функциональным блокам (например, `renderCandidates`, `renderEmployees`, `renderWizard` должны быть в отдельных файлах).

### 1.4. Хранение данных и State
- Состояние хранится в едином глобальном объекте `state` и синхронно пишется в `localStorage` при каждом изменении через `saveState()`.
- **Проблема:** При росте объема данных (сотни сотрудников и сессий) сериализация всего объекта при каждом клике приведет к фризам интерфейса.

## 2. Логика и данные

### 2.1. Модуль скоринга (`scoring.js`)
Единственный модуль, написанный чисто и изолированно.
- **Плюс:** Функции `calculateResult` и `getRecommendation` являются чистыми, легко тестируются и не зависят от глобального состояния.

### 2.2. Моковые данные
Данные генерируются функциями `seedSessions`, `seedEmployees`, и т.д.
- **Проблема:** Связи между сущностями (например, `employee.department` и список `departments`) поддерживаются строковыми совпадениями, а не ссылками по ID. Это может вызвать проблемы при интеграции с реальным backend.

## 3. Рекомендации по улучшению (Roadmap)

### Этап 1: Рефакторинг рендеринга (Низкий риск, высокий эффект)
1. **Разбить `render.js`** на папку `src/ui/pages/`, выделив каждую страницу в отдельный файл.
2. Вынести логику модальных окон (мастер оценки, фильтры) в `src/ui/modals/`.

### Этап 2: Унификация UI-кита (Средний риск)
1. Объединить `dashboard-components.js` и `dashboard-premium.js`.
2. Оставить только одну версию компонентов (Premium с префиксами `elt-`), удалив старые дубли.
3. Вынести иконки в отдельный файл `icons.js`.

### Этап 3: Декомпозиция `app.js` (Высокий риск)
1. Разделить глобальный `click` listener на смысловые контроллеры: `authController.js`, `wizardController.js`, `navigationController.js`.
2. Внедрить простую систему событий (Event Bus) или явные вызовы сервисов вместо смешивания обновления `state` и DOM-манипуляций (toast).

### Этап 4: Оптимизация State
1. Разделить `localStorage` на части: `auth`, `settings`, `data`.
2. Использовать `debounce` для `saveState()`, чтобы избежать лагов при частом обновлении (например, при вводе текста).

## Итог
Проект выглядит как быстро собранный MVP. Для перехода к Production-ready SaaS необходимо срочно остановить рост `app.js` и `render.js`, провести декомпозицию файлов и унифицировать UI-компоненты. Логика скоринга и конфигурации уже спроектирована удачно и не требует изменений.
