# ТЗ: вкладка «Вакансии» с реальными данными и интеграцией HeadHunter

## 1. Цель

Сделать вкладку «Вакансии» в Eltera полностью работающей на реальных данных:

- без демонстрационных и фейковых значений;
- с подключением аккаунта работодателя HeadHunter через OAuth;
- с возможностью импортировать/добавлять вакансии из HH в систему;
- с разграничением данных по пользователю и организации;
- с корректной работой карточек, таблиц, фильтров, периодов и drill-down-экрана;
- с использованием PostgreSQL и текущей схемы Alembic, без хранения данных на фронтенде;
- с опциональной поддержкой ИИ-бота для вопросов по вакансии и откликам.

## 2. Проблема, которую решаем

Сейчас вкладка «Вакансии» не должна опираться на заглушки, seed-данные или статические примеры. Пользователь ожидает:

- видеть только свои реальные вакансии;
- подключать свой аккаунт hh.ru и получать данные именно из него;
- выбирать, какие вакансии из HH показывать в системе;
- видеть реальные отклики, конверсии, подходящих кандидатов и статусы по выбранному периоду;
- переходить из KPI-карточек в список вакансий с уже применённым фильтром.

## 3. Текущая база и ограничения

В репозитории уже есть задел для HH-интеграции:

- модель подключения `hh_connections`;
- миграция `20260616_c8a3e5f1b2d4_hh_connections.py`;
- backend-роуты `/hh/status`, `/hh/connect`, `/hh/callback`, `/hh/disconnect`, `/hh/vacancies`;
- сервис `backend/app/services/hh.py` с OAuth, refresh token и чтением активных вакансий;
- таблица `vacancies` с полями `external_id`, `source`, `url`, `status`, `published_at`.

Также в фронтенде уже есть HH-заготовки, но они не должны использоваться как источник «фейковых» данных в продакшене.

Ключевое ограничение: текущий `GET /hh/vacancies` возвращает только активные вакансии работодателя и их текущие счётчики. Этого недостаточно для:

- исторических метрик за 7/14/30 дней;
- карточек «Закрыт за период», «Низкая конверсия», «Хорошая конверсия»;
- точной статистики откликов и смен статусов во времени.

Следовательно, для честной аналитики нужно не только читать активные вакансии, но и хранить исторические снимки/события синхронизации.

## 4. Источники данных

Единственные допустимые источники данных для вкладки:

- HeadHunter API по подключённому аккаунту работодателя;
- PostgreSQL таблицы Eltera;
- данные, созданные пользователем внутри своей организации;
- расчётные агрегаты, построенные на реальных данных HH и локальной БД.

Запрещено:

- использовать demo/seed-данные в продакшен-режиме;
- подмешивать вакансии или кандидатов из другой организации;
- показывать вакансии, не привязанные к подключённому hh-аккаунту;
- отображать данные другого пользователя без явного доступа.

## 5. Пользовательский сценарий

### 5.1 Подключение HH

Пользователь нажимает кнопку «Подключить hh.ru».

Ожидаемое поведение:

1. Фронтенд отправляет пользователя на backend endpoint подключения.
2. Backend создаёт OAuth `state`, сохраняет временное состояние подключения и редиректит на HH.
3. Пользователь проходит авторизацию на hh.ru.
4. HH возвращает callback с `code` и `state`.
5. Backend обменивает `code` на access/refresh tokens, сохраняет их и подтягивает идентичность аккаунта работодателя.
6. В интерфейсе показывается статус «Подключено» и данные аккаунта.

### 5.2 Добавление вакансий из HH

Пользователь открывает модальное окно «Добавить из HeadHunter».

Требования к модалке:

- загружает вакансии из подключённого HH-аккаунта;
- показывает только вакансии этого пользователя/работодателя;
- позволяет искать по названию;
- позволяет выбрать одну или несколько вакансий;
- после подтверждения сохраняет выбранные вакансии в систему;
- не создаёт дубли по `external_id` и организации;
- после сохранения вакансии появляются в локальной вкладке «Вакансии».

### 5.3 Работа с вкладкой

На вкладке должны отображаться только реальные данные по выбранным вакансиям и периоду.

Пользователь ожидает:

- список вакансий из своего кабинета HH;
- сводные карточки по выбранному периоду;
- таблицу эффективности вакансий;
- список подходящих кандидатов;
- таблицу/сводку откликов;
- drill-down по клику на KPI-карточки;
- переключатель периода 7 / 14 / 30 дней.

## 6. Функциональные требования

### 6.1 Авторизация и подключение HH

- Кнопка «Подключить hh.ru» должна инициировать реальный OAuth-флоу HH.
- Подключение должно выполняться только на backend.
- Токены HH нельзя хранить на клиенте.
- OAuth `state` обязателен для защиты от CSRF.
- При истечении access token должен происходить refresh через refresh token.
- При ошибке подключения нужно показывать понятную причину и статус ошибки.
- Отвязка HH должна удалять или инвалидировать локальное подключение.

### 6.2 Привязка к пользователю и организации

- Данные HH должны быть изолированы по организации.
- Данные UI должны дополнительно учитываться по текущему пользователю, если пользователь сам выбрал вакансии или если права ограничены его набором.
- Пользователь не должен видеть вакансии, не принадлежащие его организации.
- Пользователь не должен видеть вакансии, которые не были им выбраны, если включён режим персональной видимости.
- Нужно поддержать базовую роль/доступ:
  - администратор может видеть всё в своей организации;
  - рекрутер видит только доступные ему вакансии и свои выборки;
  - менеджер видит только разрешённые ему представления.

### 6.3 Импорт/синхронизация вакансий

- После подключения HH система должна уметь получать список вакансий работодателя.
- При добавлении вакансии из HH локальная запись должна связываться с `external_id` HH.
- Поля вакансии должны хранить минимум:
  - внешний ID;
  - название;
  - источник;
  - URL;
  - город;
  - статус;
  - дату публикации;
  - ссылку на пользователя, который добавил/выбрал вакансию.
- Дубли должны предотвращаться по связке `organization_id + source + external_id`.
- Если вакансия была закрыта в HH, локальный статус должен обновляться после синхронизации.

### 6.4 Период 7 / 14 / 30 дней

- Во всех ключевых блоках должен работать переключатель периода:
  - 7 дней;
  - 14 дней;
  - 30 дней.
- Переключатель должен влиять на:
  - KPI-карточки;
  - таблицу эффективности вакансий;
  - подходящих кандидатов;
  - отклики;
  - списки в drill-down;
  - любые производные графики и счётчики.
- По умолчанию использовать один предсказуемый период, согласованный с продуктом, например 7 дней.

### 6.5 KPI-карточки

Карточки должны отражать реальные данные и открывать соответствующий список по клику.

Требуемое поведение:

- «Активная вакансия без откликов»:
  - считает активные вакансии с нулём откликов за выбранный период или по текущему состоянию, в зависимости от метрики;
  - по клику открывает список таких вакансий.
- «Низкая конверсия»:
  - показывает вакансии с конверсией ниже порога;
  - по клику открывает список с сортировкой по худшему значению.
- «Хорошая конверсия»:
  - показывает вакансии с конверсией выше порога;
  - по клику открывает список с сортировкой по лучшим значениям.
- «Всего откликов»:
  - считает реальные отклики из HH/локальной синхронизации;
  - по клику открывает список вакансий с количеством откликов.
- «Всего оценок отправлено»:
  - считает реально отправленные оценки/скоринги, а не заглушки;
  - по клику открывает список вакансий и связанных оценок.
- «Подходит под профиль»:
  - учитывает только реальных кандидатов, полученных из HeadHunter;
  - не должен использовать локальных демо-кандидатов;
  - по клику открывает список подходящих кандидатов.
- «Закрытый за период»:
  - показывает вакансии, закрытые в рамках выбранного периода;
  - по клику открывает список закрытых вакансий.

### 6.6 Таблицы и drill-down

При клике на карточку или элемент таблицы пользователь должен видеть релевантный список, уже отфильтрованный по:

- периоду;
- вакансии;
- источнику;
- статусу;
- ответственному пользователю;
- конверсии/порогам;
- наличию откликов;
- источнику кандидата.

### 6.7 Таблица эффективности вакансий

Таблица должна строиться на реальных вычисляемых метриках:

- количество откликов;
- количество подходящих кандидатов;
- конверсия из отклика в подходящего;
- количество отправленных оценок;
- количество закрытых вакансий;
- время до первого отклика;
- активность вакансии;
- статус синхронизации.

Для каждой строки нужно уметь открыть карточку вакансии или список связанных кандидатов/откликов.

### 6.8 Подходящие кандидаты

- В блоке «Подходящие кандидаты» должны отображаться только кандидаты, пришедшие из HH или связанные с HH-вакансией.
- Не допускается подмешивание внутренних тестовых людей без HH-источника в эту метрику.
- Подходящий кандидат должен быть связан с:
  - вакансией;
  - источником HH;
  - пользователем/ответственным;
  - результатом оценки.

### 6.9 Отклики по вакансиям

- Отклики должны отображаться как реальные данные HH.
- Если HH отдаёт счётчики, они должны быть синхронизированы в локальную БД.
- Если доступны детали отклика, их нужно хранить и показывать в карточке кандидата/отклика.
- Любые итоги за период должны строиться по снапшотам или событиям, а не только по текущему состоянию вакансии.

## 7. Требования к данным и схеме БД

### 7.1 Использовать существующие таблицы

Минимально должны использоваться:

- `hh_connections` — подключение HH на организацию;
- `vacancies` — локальный реестр вакансий;
- `users` — текущий пользователь;
- `organizations` — контур изоляции;
- существующие таблицы кандидатов/оценок для связанных метрик.

### 7.2 Что нужно добавить

Чтобы требования по периоду и drill-down работали корректно, нужна история.

Рекомендуемые сущности:

- `hh_vacancy_selections`
  - связь пользователь ↔ выбранная HH-вакансия;
  - хранит, кто добавил вакансию в свой список;
  - позволяет показывать вакансии только выбранным пользователем.
- `hh_vacancy_snapshots`
  - ежедневные или периодические снимки метрик вакансии;
  - хранит счётчики откликов, закрытий, статусов и агрегаты;
  - используется для 7/14/30 дней.
- `hh_response_events` или аналогичная таблица событий
  - хранит поступающие отклики/изменения статусов;
  - позволяет строить честные отчёты по периоду.
- при необходимости `hh_candidate_links`
  - связывает отклик HH с локальным кандидатом/человеком.

Если команда решит не добавлять отдельные таблицы, то нужно явно описать, каким образом историчность и персональная видимость реализуются на уже существующих сущностях. Без истории требования по периодам не выполнить корректно.

### 7.3 Поля безопасности и целостности

- внешние токены HH должны храниться только на backend;
- желательно шифровать токены в БД;
- OAuth state должен быть одноразовым;
- все записи должны иметь `organization_id`;
- все выборки должны фильтроваться по `organization_id`;
- если реализована персональная видимость, дополнительно фильтровать по `user_id`.

## 8. Backend API

### 8.1 Существующие endpoint'ы

- `GET /hh/status`
- `GET /hh/connect`
- `GET /hh/callback`
- `POST /hh/disconnect`
- `GET /hh/vacancies`

### 8.2 Что требуется добавить

Для полноценной вкладки нужны дополнительные endpoint'ы или расширение существующих:

- список вакансий с фильтрами;
- импорт/добавление вакансии из HH;
- список выбранных пользователем вакансий;
- детализация по вакансии;
- KPI по периоду;
- список откликов;
- drill-down по карточкам;
- данные для ИИ-бота по вакансии и откликам.

Примеры логики:

- `GET /vacancies?period=7d&status=active&source=hh`
- `POST /vacancies/import/hh`
- `GET /vacancies/{id}/metrics?period=14d`
- `GET /vacancies/{id}/responses?period=30d`
- `GET /vacancies/{id}/candidates?fit=true`

Точные URI можно согласовать отдельно, но поведение должно остаться таким же.

## 9. Frontend требования

### 9.1 Статус HH

- кнопка «Подключить hh.ru» должна менять состояние интерфейса в зависимости от реального статуса backend;
- после успешного подключения интерфейс должен обновляться без ручной перезагрузки;
- при ошибке показывать текст с причиной;
- должна быть кнопка отключения.

### 9.2 Модалка добавления вакансии

В модальном окне должны быть:

- список вакансий HH;
- поиск;
- фильтр по статусу;
- чекбоксы или единичный выбор;
- кнопка подтверждения;
- состояние загрузки;
- состояние пустого списка;
- обработка ошибки API.

### 9.3 Отображение карточек и таблиц

- все KPI-карточки должны быть кликабельными;
- клики должны приводить к списку/модальному окну с фильтрацией;
- данные должны обновляться при смене периода;
- состояние фильтров должно сохраняться на уровне страницы.

### 9.4 Удаление фейковых данных

- убрать demo-подстановки для вкладки «Вакансии» в продакшен-режиме;
- если нужен dev-fallback, он должен быть явно отделён от production и недоступен при подключённом backend;
- пустое состояние должно быть честным: если данных нет, показываем пустую страницу, а не генератор тестовых вакансий.

## 10. Реальные правила расчётов

### 10.1 Конверсия

Конверсия вакансии должна считаться из реальных событий.

Вариант базовой формулы:

- `conversion = suitable_candidates / responses * 100`

Если команда использует другую формулу, её нужно зафиксировать в одном месте и не менять между карточками и таблицей.

### 10.2 Подходит под профиль

Считаются только реальные кандидаты, пришедшие из HH или привязанные к HH-отклику.

### 10.3 Закрытые вакансии

Вакансия считается закрытой за период, если её статус изменился на `closed` в этом периоде или если HH прислал событие закрытия/архивации.

### 10.4 Всего откликов

Считаются только реальные отклики HH, синхронизированные в локальную БД.

## 11. Безопасность

Обязательные требования:

- OAuth только через backend;
- state-параметр обязателен;
- access/refresh tokens не отдавать фронту;
- токены хранить под организацией;
- все запросы должны выполняться с учётом текущего пользователя и организации;
- не допускать утечек вакансий между организациями;
- логировать ошибки без утечки секретов;
- не хранить чувствительные данные в URL, кроме стандартного OAuth callback.

Рекомендуемые требования:

- шифрование токенов в БД;
- audit trail для подключения/отключения HH;
- rate limiting на синхронизацию;
- фоновые задачи для периодической актуализации.

## 12. ИИ-бот, опционально

Опционально, но желательно:

- бот может отвечать на вопросы о конкретной вакансии;
- бот может кратко объяснять динамику откликов;
- бот может показывать причину низкой конверсии;
- бот может подсказать, что делать с вакансией на основе реальных данных.

Ограничения:

- бот должен отвечать только по данным текущей организации;
- бот не должен использовать фейковые данные;
- бот должен видеть только ту вакансию, на которую есть права у пользователя;
- если данных недостаточно, бот должен честно говорить об этом.

## 13. Критерии приёмки

Фича считается готовой, если:

1. Кнопка «Подключить hh.ru» реально запускает OAuth HH и подключает аккаунт.
2. В интерфейсе отображается только реальная информация из HH и БД.
3. Вакансии можно добавить через модалку из HH.
4. Вкладка не использует seed/demo-данные в рабочем режиме.
5. Карточки «Активная вакансия без откликов», «Низкая конверсия», «Хорошая конверсия», «Всего откликов», «Всего оценок отправлено», «Подходит под профиль», «Закрытый за период» открывают корректные списки.
6. Переключатель 7 / 14 / 30 дней влияет на все расчёты и списки.
7. Реализовано разграничение по организации, а при необходимости и по пользователю.
8. Для периодов и закрытий используются реальные исторические данные, а не только текущее состояние вакансии.
9. Ошибки HH обрабатываются без падения страницы и без утечки токенов.
10. Документация по API и схеме данных обновлена.

## 14. Что не входит в MVP

- расширенная аналитика на уровне LTV и прогнозов;
- сложная multi-tenant RBAC-модель с кастомными правами на уровне полей;
- импорт из других джоб-сайтов, если не согласовано отдельно;
- глубокая LLM-аналитика по всей воронке без отдельного согласования по данным и доступам.

## 15. Риски и зависимости

- HH может ограничивать доступ по приложениям, модерации и scopes;
- без исторических снимков невозможно честно показать период 7/14/30;
- если в системе одна организация на несколько пользователей, нужно заранее определить правило видимости выбранных вакансий;
- расчёт конверсий нужно зафиксировать до разработки, иначе цифры на карточках и в таблице могут разъехаться.

## 16. Рекомендуемый следующий шаг

Согласовать:

- точную формулу конверсии;
- правила видимости вакансий между пользователями;
- список новых таблиц для историчности;
- набор API endpoint'ов для drill-down;
- поведение модалки «Добавить из HeadHunter».
