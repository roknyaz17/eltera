# Eltera — дорожная карта

## ✅ Сделано


### Подбор и кандидаты
- Раздел «Кандидаты» как мини-ATS: список, KPI, воронка, тепловая карта, источники.
- Добавление кандидата, отправка оценки из меню действий, просмотр ответов, PDF-отчёт.
- Импорт кандидатов из Excel (шаблон, дедупликация, отчёт об ошибках).

### Сотрудники и структура
- Раздел «Сотрудники»: список, KPI, риски, удовлетворённость, карточка с результатами по каждому тесту.
- Импорт сотрудников из Excel (ФИО, контакты, отдел, руководитель, дата выхода и т.д.); руководитель связывается по ФИО; новым с датой выхода сразу заводится цикл адаптации.
- Неоценённые сотрудники/кандидаты корректно показываются как «Не оценён», риск «—» (не искажают аналитику).
- «Структура»: список сотрудников + граф по отделам (масштаб, поиск, клик по карточке → профиль).

### Конструктор оценок
- Двухуровневая модель: **профиль = набор компетенций**, у компетенции — свой банк вопросов (переиспользуется во всех профилях).
- Два раздела: «Профили» (сборка из компетенций) и «Компетенции» (банк вопросов: один/несколько вариантов, шкала, открытый с AI-эталоном).
- Импорт базы из одного Excel (3 листа: Компетенции / Вопросы / Профили), идемпотентно.
- Сервисные тесты (360 / Performance / Адаптация) защищены от удаления.
- «Инструменты → Профили» — каталог готовых профилей из БД: фильтры, запуск оценки, открытие в конструкторе.

### Оценка и прохождение
- Мастер «Создать оценку» (точечно/группой/по отделу), оценочные ссылки, раздел «Оценки» с фильтрами и типом получателя.
- Прохождение теста: стартовый экран (ФИО/почта) + пошаговый мастер «один вопрос — один экран» с прогрессом и таймером.

### Адаптация
- Полный цикл онбординга: каденция 1/3/7/14/30/60/90/180 дней, авто-старт по дате выхода, ежедневный «тик» (APScheduler), напоминания, просрочки, закрытие цикла.
- Таймлайн циклов, сигналы руководителю (риск/пропуск), воронка «Этапы адаптации» на реальных данных, просмотр ответов по этапу.
- Данные обновляются при заходе на вкладку (без ручного «Обработать дозревшие»).

### 360° и Performance Review
- 360°: оцениваемый + оценщики по ролям (руководитель/коллега/подчинённый/самооценка), разные опросы по ролям, сводный отчёт (самооценка vs внешняя, расхождения).
- Performance Review: вкладка на реальных данных, 9-box.

### Главная и уведомления
- «Главная» — сводный дашборд по всем разделам: KPI, «Требует внимания», лента событий, ближайшие опросы адаптации.
- Центр уведомлений: колокольчик с непрочитанными, панель событий, отметка прочитанного, фоновое обновление.

### Коммуникации
- **Email (рабочий SMTP, Reg.ru):** приглашения на оценку/адаптацию/360, напоминания, письмо HR «оценка пройдена / отчёт готов», welcome-письмо при импорте.
- **Поддержка:** обращения уходят в Telegram-бота (текст зависит от типа обращения).

## 💰 Себестоимость оценки

Переменная стоимость одной оценки — это **только AI-токены**. Всё остальное бесплатно/фиксировано:

- **Скоринг открытого вопроса** — 1 вызов LLM на каждый открытый вопрос (`ai_scoring.score_open_answer`, вызывается в цикле прохождения только для `QuestionType.open`). Шкальные и закрытые вопросы считаются арифметикой → **₽0**.
- **Нарратив отчёта** — 1 вызов LLM, **только для кандидатов** (`report_jobs.generate_and_store_narrative` → `get_candidate` возвращает `None` для сотрудников/360/адаптации). Результат кэшируется в сессии.
- **PDF-отчёт** — `reportlab`, рендерится локально из уже посчитанных баллов и сохранённого нарратива. Без AI и сети → **₽0** (перегенерация/перекачка бесплатна).
- **Email/SMTP, хостинг** — фиксированные, на одну оценку ничтожны.

**Модель и тариф:** DeepSeek V4 Flash, **₽34.7 / 1M токенов** (вход = выход).
Формула: `стоимость = (токены_вход + токены_выход) × 34.7 / 1 000 000`.

Оценка токенов на вызов (вход — оценочный, зависит от длины вопроса/эталона/ответа; выход ограничен `max_tokens`):

| Компонент | Токены (≈) | Стоимость |
|---|---|---|
| Скоринг 1 открытого вопроса | ~650 (350 + 300) | ₽0.023 |
| Нарратив отчёта (кандидат) | ~2 100 (300 + 1800) | ₽0.073 |
| Шкала/выбор (любое количество) | 0 | ₽0 |
| PDF-отчёт | 0 | ₽0 |

**Стоимость одной оценки по типам** (при ~1 открытом вопросе на тест):

| Тип | Из чего | Стоимость |
|---|---|---|
| Кандидат | 1 скоринг + нарратив + PDF | **₽0.095** |
| Сотрудник (компетенции) | 1 скоринг + PDF | **₽0.023** |
| Performance Review | 1 скоринг + PDF | **₽0.023** |
| 360° — 1 участник | 1 скоринг | **₽0.023** |
| 360° — полный (оцениваемый + 5 оценщиков) | 6 скорингов | **₽0.135** |
| Адаптация — 1 этап | 1 скоринг | **₽0.023** |
| Адаптация — полный цикл (8 этапов) | 8 скорингов | **₽0.180** |

**Самый дорогой профиль в текущей БД — «Оператор call-центра»** (кандидат, 20 вопросов, 2 открытых):
2 скоринга (₽0.045) + нарратив (₽0.073) + PDF (₽0) = **≈ ₽0.12**.
То есть 1 000 таких прогонов ≈ ₽120, 10 000 ≈ ₽1 200.

> ⚠️ Стоимость **линейна по числу открытых вопросов**; драйверы цены — (а) количество открытых вопросов, (б) факт кандидата (только у него генерится нарратив).
> ⚠️ `AI_ENABLED=true` (боевой режим) — DeepSeek вызывается на каждом прохождении, цифры выше — реальная себестоимость. Точную среднюю по токенам уточним по факту прогонов (вход зависит от длины ответов).

## 🔧 В работе

Незакрытые задачи, которые нужно завершить до релиза.

- **Интеграция с HeadHunter.** OAuth-флоу, хранение/обновление токенов и чтение вакансий с откликами реализованы; вкладка «Вакансии» подключается через «Подключить hh.ru». Ожидается: регистрация приложения в `dev.hh.ru` (на модерации) и `client_id/secret`; после получения — живая проверка и точечная донастройка маппинга ответов HH (`employer_id` из `/me`, счётчики откликов).
- **Публичный адрес (деплой-преблокер).** Сейчас `APP_PUBLIC_URL = localhost` → ссылки в письмах открываются только локально. Нужен боевой домен (HTTPS), чтобы письма, демо-доступ и HH-redirect работали у всех.
- **Запуск оценки по профилю кандидата.** Сейчас «Запустить оценку» из каталога профилей ведёт в мастер в режиме сотрудника; для кандидатских профилей нужно развести на флоу «Создать ссылку кандидату» по `target_type`.
- **«Соответствие должности» vs Performance Review.** Развести логику: соответствие — из профильной оценки компетенций; результативность/риск — из Performance Review (отдельное поле). Сейчас на паузе.

## 🕳️ Пробелы, которые нужно закрыть

Важное, чего пока нет (часть — за рамками MVP, но нужно для прод-эксплуатации).

- **Аутентификация и роли.** Сейчас один демо-вход. Нужны реальный логин и роли по тарифам с разными правами и областью видимости.
- **Мультиарендность.** Модель пока одно-организационная; для нескольких компаний/холдингов нужно разграничение данных по организациям.
- **PWA.** Установка как приложение (иконка на телефон/рабочий стол) — обсуждалось, не сделано.
- **AI-генерация компетенций/вопросов** по описанию роли — обсуждалось, не сделано.
- **Кандидаты из откликов HH** (`/negotiations`) — подтягивание откликнувшихся в раздел «Кандидаты» (после базовой HH-интеграции).
- **Деплой и инфраструктура.** Боевой домен + HTTPS, переезд с SQLite на PostgreSQL для нагрузки, секреты (SMTP/HH/Telegram) в окружении, авто-перезапуск бэкенда.
- **Модерация HH** для прод-доступа к данным сторонних работодателей.
- **Автотесты** ключевых сценариев (оценка, адаптация, импорт) для защиты от регрессий.
- **Невалидное поведение некоторых карточек на дашборде** сейчас при нажатии на карточку в дашборде мы можем получать не те данные, которые ожидает пользователь. (Например: При нажатии на карточку "Отправлено ссылок" вместо ожидаемых ссылок мы видим список кандидатов.)
